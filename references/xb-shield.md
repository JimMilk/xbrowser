# xbrowser Shield 网络防护 — 设计文档

## 目的

xbrowser 控制的是用户真实浏览器，会带上用户已登录网站的 cookie 和会话状态。
Shield 网络防护默认启用，用于降低 prompt injection 诱导 agent 访问内网资源、云元数据端点或本机敏感页面的风险。

---

## 架构概览：4 层纵深防御

Shield 采用分层拦截模型，从最外层到最内层逐级收紧：

```
用户访问请求
  │
  ▼
┌──────────────────────────────────────────────┐
│ Layer 0: 始终启用的预检                       │
│  • 云元数据 IP 地址（硬编码，永不放行）         │
│  • 危险协议（file://、data://、ftp:// 等）     │
│  • URL 格式校验                               │
│  拦截原因: INVALID_FORMAT、DANGEROUS_PROTOCOL、 │
│           CLOUD_METADATA                      │
└──────────────────────────────────────────────┘
  │ 通过
  ▼
┌──────────────────────────────────────────────┐
│ Layer 1: 私有网络检测                         │
│  • RFC 1918 私有地址（10.x, 172.16-31, 192.168）│
│  • 链路本地地址（169.254.0.0/16，不含 169.254.169.254）│
│  • 回环地址（127.0.0.0/8, ::1）               │
│  • 运营商级 NAT（100.64.0.0/10）              │
│  • IPv6 本地地址（fe80::/10, fc00::/7, ::ffff:0:0）│
│  • IPv4-mapped IPv6 私有地址                  │
│  • .local / .localhost / .internal 域名       │
│  拦截原因: PRIVATE_NETWORK                    │
└──────────────────────────────────────────────┘
  │ 通过（域名需要 DNS 解析）
  ▼
┌──────────────────────────────────────────────┐
│ Layer 2: DNS 重绑定检测                       │
│  • 解析域名后检查所有返回的 IP                 │
│  • 若解析到私有/内网地址 → 拦截                │
│  • 若解析到云元数据 IP → 拦截                  │
│  拦截原因: PRIVATE_NETWORK_VIA_DNS、           │
│           CLOUD_METADATA_VIA_DNS              │
└──────────────────────────────────────────────┘
  │ 通过
  ▼
┌──────────────────────────────────────────────┐
│ Layer 3: 用户白名单                           │
│  • 用户可以明确确认将特定 host:port 加入白名单   │
│  • 白名单条目经过 HMAC 签名保护，防止篡改       │
│  • 云元数据端点 **永不加入白名单**（硬编码拒绝） │
│  • 危险协议 **永不加入白名单**                  │
└──────────────────────────────────────────────┘
  │ 允许访问
  ▼
    正常导航
```

**关键设计原则**：
- Layer 0 的云元数据拒绝是**硬编码的**，代码层面不可绕过
- 每层独立判断，任何一层拦截都会阻止访问
- 白名单只影响 Layer 1 和 Layer 2 的私有网络拦截，不影响 Layer 0

## 完整的拦截原因枚举

所有拦截原因定义在 `ShieldReason` 常量中：

### INVALID_FORMAT

**含义**：URL 无法被 `new URL()` 解析，或 hostname 为空。

**触发条件**：
- 传入的 URL 字符串不是合法的绝对 URL
- 解析后 hostname 为空字符串
- URL 中缺少协议部分

**可以加入白名单**：否 — 格式错误不是安全决策，是输入校验。

**用户可见说明**：URL 格式不正确，请检查输入。

---

### DANGEROUS_PROTOCOL

**含义**：URL 使用了不允许的协议 scheme。

**被阻止的协议完整列表**：

| 协议 | 风险说明 |
|------|----------|
| `file://` | 读取本地文件系统 |
| `chrome://` | 访问 Chrome 内部页面和设置 |
| `chrome-extension://` | 访问浏览器扩展内部资源 |
| `javascript:` | 执行任意 JavaScript |
| `data:` | 内联数据（可包含可执行内容） |
| `view-source:` | 查看页面源码（可能泄露敏感信息） |
| `about:` | 访问浏览器内部 about 页面 |
| `ftp://` | 文件传输协议 |
| `gopher://` | 已弃用的 Gopher 协议 |

**允许的协议**：仅 `http:` 和 `https:`。

**可以加入白名单**：否 — 危险协议永不通过。

**用户可见说明**：使用了不被允许的协议，该 URL 不可访问。

---

### PRIVATE_NETWORK

**含义**：URL 解析出的 IP 地址属于私有网络范围。

**被拦截的 IP 范围**：

| IP 范围 | CIDR | 说明 |
|---------|------|------|
| `0.0.0.0/8` | /8 | "This" network（含 0.0.0.0） |
| `10.0.0.0/8` | /8 | RFC 1918 私有网络 A 类 |
| `127.0.0.0/8` | /8 | 回环地址（localhost） |
| `169.254.0.0/16` | /16 | 链路本地地址（含云元数据但在此层检查前已过滤） |
| `172.16.0.0/12` | /12 | RFC 1918 私有网络 B 类 |
| `192.168.0.0/16` | /16 | RFC 1918 私有网络 C 类 |
| `100.64.0.0/10` | /10 | 运营商级 NAT（RFC 6598） |
| `224.0.0.0/4` | /4 | 组播地址 |
| `::1` | /128 | IPv6 回环 |
| `::` | /128 | IPv6 未指定地址 |
| `fe80::/10` | /10 | IPv6 链路本地 |
| `fc00::/7` | /7 | IPv6 唯一本地地址（ULA） |
| `::ffff:0:0/96` | /96 | IPv4-mapped IPv6 地址（仅当映射的 IPv4 为私有时） |

**特殊域名匹配**：
- 精确匹配 `localhost`（不区分大小写）
- `.localhost` 子域名后缀
- `.local` 子域名后缀（mDNS 域）
- `.internal` 子域名后缀

**IP 格式检测**：
- 类似 `0` 开头的八位组（如 `010.0.0.1`）也被检测为私有网络

**可以加入白名单**：**是** — 这是用户主动允许的操作。

**白名单格式**：`host:port`（如 `192.168.1.10:8080`），匹配时不区分大小写，支持 IPv6 的 `[address]:port` 格式。

---

### PRIVATE_NETWORK_VIA_DNS

**含义**：URL 的主机名不是 IP 地址而是域名，经过 DNS 解析后，所有解析结果中的 IP 地址属于私有网络范围。

**触发条件**：
1. URL 解析出的 hostname 不是 IP 地址（需要 DNS 查询）
2. DNS 查询返回的所有 A/AAAA 记录中，**至少有一个**是私有 IP
3. 该 host:port 不在白名单中

**可以加入白名单**：**是** — 前提是用户明确了解 DNS 重绑定风险。

**风险说明**：这是 DNS 重绑定的典型攻击模式。域名在安全检查时解析到公网 IP，但在后续请求时被重新绑定到内网 IP。xbrowser 的 DNS 检查要求**所有**解析到的 IP 都是公网地址才放行。

---

### CLOUD_METADATA

**含义**：URL 的 IP 地址或主机名指向云平台元数据服务端点。

**被阻止的 IP 地址**：

| IP | 平台 | 说明 |
|----|------|------|
| `169.254.169.254` | AWS / GCP / Azure / 阿里云 / 腾讯云 / 华为云 | 链路本地元数据端点 |
| `169.254.170.2` | AWS ECS | ECS 任务元数据端点 |
| `100.100.100.200` | 阿里云 | VPC 内元数据端点 |

**被阻止的主机名**：

| 主机名 | 平台 |
|--------|------|
| `metadata.google.internal` | GCP Compute Engine |
| `metadata.azure.com` | Azure Instance Metadata Service |
| `metadata` | 通用元数据主机名（纯 `metadata` 被 DNS 解析后可指向任意 IP） |

**可以加入白名单**：**永不**。这是硬编码的拒绝，代码层面不可绕过。即使云元数据 IP 经过 DNS 解析（CLOUD_METADATA_VIA_DNS），也同样不可绕行。

**设计原理**：云元数据端点通常不需要认证即可泄露实例凭证、临时密钥、IAM 角色凭证等敏感信息。在 prompt injection 场景下，攻击者可能诱导 agent 访问 `http://169.254.169.254/latest/meta-data/iam/security-credentials/` 获取 AWS 凭证。这条路径必须永久关闭。

**用户可见说明**：云元数据端点不可访问，这是安全策略的硬性底线。

---

### CLOUD_METADATA_VIA_DNS

**含义**：经过 DNS 解析后，域名解析到的 IP 中包含云元数据端点 IP。

**触发条件**：
1. DNS 查询返回的 IP 列表中**至少有一个**是云元数据 IP
2. 这种情况可能是被篡改的 DNS 记录或 hosts 文件注入

**可以加入白名单**：**永不** — 与 CLOUD_METADATA 同级处理。

---

### DNS_RESOLVE_FAILED

**含义**：域名 DNS 解析失败（NXDOMAIN、SERVFAIL、超时等）。

**触发条件**：
1. URL 解析出的 hostname 不是 IP 地址
2. `dns.lookup()` 抛出异常

**可以加入白名单**：否 — 这是网络连通性问题，不是安全决策。

**用户可见说明**：DNS 解析失败，目标地址无法访问。

---

## 白名单流程

### 完整生命周期

```
用户访问被拦截的私有地址
    │
    ▼
Agent 内部执行 xb guide shield-allow <host:port>
    │
    ▼
写入 pending 文件（HMAC 签名，带时效性）
    │
    ▼
返回 awaits_user_input: true 的决策点
    message: "网络防护拦截了 <host:port>。是否允许访问该地址？"
    options: [取消(推荐), 确认加入白名单]
    │
    ▼
用户明确确认（必须包含：操作 + 对象 + 确认语义）
    │
    ▼
Agent 内部执行 xb shield allow <host:port>
    │
    ▼
读取 pending 文件，验证 HMAC 签名
    │
    ▼
签名验证通过 → 加入白名单配置 → 写入 HMAC 签名的配置文件
    │
    ▼
Agent 重新导航到目标地址 → 白名单匹配 → 放行
```

### HMAC 签名机制

白名单配置文件使用 HMAC-SHA256 签名防止篡改：

1. **机器 ID 派生**（平台相关）：
   - macOS：`ioreg` 读取 `IOPlatformUUID`
   - Linux：读取 `/etc/machine-id` 或 `/var/lib/dbus/machine-id`
   - Windows：查询注册表 `HKLM\SOFTWARE\Microsoft\Cryptography\MachineGuid`
   - 回退：`SHA256(hostname|username|homedir)`

2. **密钥派生**：`HMAC-SHA256(machineId, "xbrowser-shield-hmac-v1")`

3. **签名**：`HMAC-SHA256(key, canonicalize(config))`

4. **验证**：使用 `timingSafeEqual` 防止时序攻击

5. **规范化**：排序键、递归处理嵌套对象、排除 `_sig` 字段

**安全性**：
- 配置文件被篡改 → 签名验证失败 → 重置为安全默认值 + 记录日志
- Pending 文件被篡改 → 签名验证失败 → 拒绝加白
- 跨机器复制配置文件无效（machine ID 不同）

### Pending 确认流程

加白不是直接生效的——它通过 pending 文件实现"用户二次确认"：

1. `xb guide shield-allow <host:port>` 执行时：
   - 先验证 host:port 格式合法性（`validateAllowlistEntry`）
   - 检查 host 是否为云元数据端点（硬编码拒绝）
   - 写入 HMAC 签名的 pending 文件到 `shield/pending/` 目录
   - 返回决策点 JSON 给 agent

2. 用户确认后，agent 执行 `xb shield allow <host:port>` 时：
   - 读取对应的 pending 文件
   - 验证 HMAC 签名
   - 验证 host:port 是否与 pending 中的一致
   - 通过 → 写入白名单配置
   - 记录 `ALLOW_ADDED` 日志

3. 关闭防护也走同样的 pending 流程：
   - `xb guide shield-off` → 写入 pending 文件 → 用户确认 → `xb shield disable YYYYMMDD` → 验证 → 关闭

### 为什么云元数据永远不可白名单化

```
xb guide shield-allow 169.254.169.254:80
  → 格式校验失败 → 返回错误 "云元数据端点不可加入白名单"
  → 不会写入 pending 文件
  → 不会返回决策点
  → 没有任何后续路径
```

这是**代码层面**的硬拒绝，在所有用户交互之前就拦截了。没有任何用户确认、配置修改、或绕过路径。

## 配置存储

Shield 配置存储在 `~/.openclaw/tools/xbrowser/shield/config.json`：

```json
{
  "version": 1,
  "enabled": true,
  "allowlist": ["192.168.1.10:8080", "10.0.0.5:3000"],
  "_sig": "hex-encoded-hmac-signature"
}
```

- `_sig` 由 HMAC 签名保护
- 配置损坏（解析失败或签名不匹配）→ 重置为安全默认值（启用 + 空白名单）+ 记录 `CONFIG_CORRUPTED` 日志
- `init` 时自动初始化 shield 配置（如果不存在）

## 日志系统

Shield 日志存储在 `~/.openclaw/tools/xbrowser/shield/logs/protection.jsonl`：

### 日志类型

| 类型 | 说明 | 触发 |
|------|------|------|
| `BLOCK` | 访问被拦截 | 任何 URL 被 shield 拒绝 |
| `OPEN_ERROR` | 打开错误（非安全拦截） | INVALID_FORMAT 或 DNS_RESOLVE_FAILED |
| `SHIELD_INITIALIZED` | 防护初始化 | 首次创建或版本迁移 |
| `SHIELD_ON` | 防护启用 | `xb shield enable` |
| `SHIELD_OFF` | 防护关闭 | 用户确认后关闭 |
| `ALLOW_ADDED` | 白名单添加 | 用户确认后加入白名单 |
| `ALLOW_REMOVED` | 白名单删除 | `xb shield remove` |
| `CONFIG_CORRUPTED` | 配置损坏已修复 | 配置文件签名无效 |

### 日志特性

- JSONL 格式，每行一条记录
- 自动轮转：保留最近 3000 条（`rotateThreshold`），每 100 条写入时检查一次（`rotateCheckEvery`）
- 文件上锁写入（`.lock` 文件，100ms 超时），防止并发写入损坏
- `xb shield logs` 返回最近记录，默认 20 条，上限 1000 条

## 与 Playwright MCP 的比较

| 维度 | Playwright MCP | xbrowser Shield |
|------|---------------|----------------|
| **防护层级** | 1 层：`--allowed-hosts` 黑/白名单 + `--blocked-origins` 域名拦截 | 4 层纵深防御 |
| **云元数据保护** | 无专门保护，依赖用户自行配置 `--allowed-hosts` | 硬编码阻止，永不绕行 |
| **DNS 重绑定** | 无检测 | Layer 2 专有检测，解析后二次验证 IP |
| **危险协议** | `file://` 默认允许（`--allow-file-access` 控制） | 硬编码拒绝所有非 http/https 协议 |
| **签名保护** | 无 | HMAC-SHA256 签名，防篡改 |
| **日志审计** | 无专门安全日志 | 完整的结构化日志，含拦截原因和类型 |
| **用户决策** | 无人在回路 | 每项白名单都需要用户明确确认 |
| **永不绕行的概念** | 无 — 任何规则都可通过 CLI 参数覆盖 | 云元数据和危险协议是硬编码的永久拒绝 |

**核心差异**：PMCP 依赖用户（或 AI agent）正确配置安全参数，xbrowser Shield 采用"安全默认"策略 — 任何未明确允许的都是阻止的，且某些类别（云元数据、危险协议）在任何情况下都不能放行。

## 用户可见回复规则

当用户询问"我能做什么 / 安全流程 / 注意事项"，或浏览器访问返回网络防护拦截错误时，最终回复必须使用自然语言说明流程，不得把 raw 命令作为用户操作步骤输出；包括状态、白名单、日志、引导、放行、撤销、运行、版本等所有 xb CLI 命令。

面向用户的安全流程：

1. 将拦截识别为安全策略，而不是浏览器故障或网络故障。
2. 说明被拦截的原因和访问该地址的风险。
3. 告知用户不会绕过网络防护，也不会代用户做安全决策。
4. 如果目标允许临时放行，要求用户明确确认操作和对象；确认后由 agent 内部执行对应操作。
5. 不主动建议关闭整个防护；只有用户明确要求关闭整体防护时，才说明高风险并发起确认。
6. 对云元数据端点、危险协议等不可放行目标，直接告知用户该 URL 不可访问。

禁止的最终回复形态：

- 不要说"你可以执行/请执行"后跟 raw 命令。
- 不要用具体 CLI 命令描述"查询状态/查看白名单/查看日志/排查原因"；改用"我会在内部查询最近的防护记录/拦截记录"。
- 加白、继续访问场景不要提到撤销白名单；撤销语义只用于用户明确要求撤销已有白名单或清理测试状态。
- guide / shield / status / list / logs / remove / enable / run / version 等 CLI 术语只用于内部实现，不是用户手动步骤。

## 拦截后的 agent 内部行为

CLI 细节只作为 internal/implementation reference。需要具体命令时查阅 [xb-cli-commands.md](./xb-cli-commands.md)，但不得在最终用户回复中展开：

1. 对可放行的 `host:port`，agent 内部发起加白确认流程。
2. guide 返回等待用户输入时，agent 展示自然语言风险说明和选项，停止等待用户选择。
3. 用户明确确认后，agent 只执行当前返回映射中明确给出的对应内部命令。
4. 用户要求关闭整体防护时，agent 内部发起关闭防护确认流程，但不得代用户确认。
5. 撤销已有白名单或清理测试状态时，agent 内部执行撤销操作；不要把撤销描述成加白或继续访问方式。

## 操作语义边界

Shield 相关命令分为"公开管理命令"和"确认引导命令"。agent 只能按每个命令的真实语义解释结果，不能因为用户想达成某个效果，就把一个操作说成相反含义。

| 操作 | 正确语义 | 禁止表述 |
|------|----------|----------|
| 临时放行 | 进入用户确认流程；用户明确确认后，才加入白名单 | 不得说成撤销白名单，不得说成已经放行 |
| 撤销放行 | 删除已有白名单条目 | 不得说成加白、放行或允许访问 |
| 关闭防护 | 进入关闭防护确认流程；用户明确确认后，才关闭 | 不得在模糊授权下关闭 |
| 启用防护 | 恢复/打开安全防护 | 不得说成关闭防护 |
| guide | 返回给最终用户的确认引导和选项 | guide 返回不代表操作已完成，不得说成已经加白或已经关闭 |

硬规则：撤销不是加白；启用不是关闭；guide 只是确认引导，不代表操作完成；不要把任何 shield 操作解释成相反语义。CLI/agent 执行参考不是最终用户回复模板，面向用户时必须用自然语言说明"需要确认/等待选择"，而不是把映射命令当成用户已经确认。

## 强确认边界

加白名单、关闭防护会改变网络安全边界，必须等待用户明确确认。有效确认必须同时具备三项：

1. **明确操作**：例如"临时放行""加入白名单""关闭防护"。
2. **明确对象**：加白时必须指明 `host:port`；关闭防护时必须指明整体网络防护。
3. **明确确认语义**：例如"我确认""我了解风险并确认"。

以下表达都不构成确认，不能触发加白或关闭防护：

- "你自己看着办"
- "按你推荐"
- "随便"
- "别问我"
- "你决定"
- "先试试"
- "应该可以吧"
- "可以的话就做"

可接受的确认示例：

- "我确认临时放行 127.0.0.1:65531"
- "我确认把 192.168.1.10:8080 加入白名单"
- "我了解风险，确认关闭防护"

不可接受的确认示例：

- "按你推荐处理"——没有明确确认语义。
- "可以的话就放一下"——条件式授权，不是明确确认。
- "你决定要不要关闭"——把安全决策委托给 agent，不是用户确认。

即使 guide 返回了确认映射，也只有在用户的自然语言回复满足强确认边界后，agent 才能按当前映射执行对应内部命令。

## 面向用户的白名单流程

白名单用于允许特定 `host:port`，必须由用户明确确认。最终回复用自然语言说明：

1. 当前访问被安全策略拦截。
2. 该地址属于可申请临时放行的范围，并说明放行会允许浏览器访问该地址。
3. 要求用户明确确认"临时放行 + 具体 host:port"。
4. 用户确认后，由 agent 内部完成放行并继续访问。
5. 如用户之后要求撤销或清理测试状态，再用自然语言说明可撤销/移除该白名单。

最终回复不得把 raw 命令列为用户操作步骤。

## 面向用户的关闭防护流程

关闭整个防护风险很高，强烈不推荐。最终回复用自然语言说明：

1. 不主动建议关闭整体防护。
2. 如果用户只是想访问一个特定地址，优先说明"可对该地址单独发起临时放行确认"。
3. 只有用户明确要求关闭整体防护时，才说明风险并要求明确确认。
4. 用户确认后，由 agent 内部发起并执行关闭防护的确认流程。

最终回复不得把 raw 命令当作用户手动步骤展示。

## 用户确认边界

agent 不应根据已有命令名规律推断、构造或补全后续操作命令。需要用户确认的操作，唯一合法来源是当前 guide 返回的用户选择映射。

正确做法：

1. 执行内部 guide 工具。
2. 展示返回的决策点。
3. 等用户明确选择。
4. 只执行当前用户选择映射中明确给出的内部命令。

## 日志与排查

日志与状态查询是 agent 内部排查手段，不是最终用户回复模板。用户询问安全流程时，不要要求用户手动执行查询命令，也不要在"我会..."句式里展示具体 CLI 命令。

- status 用于 agent 内部确认防护是否启用、白名单数量和 24 小时拦截次数。
- list 用于 agent 内部确认某个地址是否已在白名单中。
- logs 用于 agent 内部查看最近拦截和防护相关事件。
- 面向用户描述排查动作时，写"我会在内部查询最近的防护记录/拦截记录"，不要写具体命令。
